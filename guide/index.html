GCP Nexus — Tester's Guide

A walkthrough of every screen and action in the GCP / GCPC request portal — from submitting a request to engagements, review decisions, signatures and letters. Use it to plan and run your test passes.

Audience: UAT / QA testers Covers: GCP & GCPC channels Read-time: ~25 min

00Overview

GCP Nexus digitises the end-to-end handling of contract-procurement requests for the Group Contract Procurement (GCP) and the Group Contract Procurement Committee (GCPC). It replaces Word-based forms with guided web forms and routes every request through a structured review lifecycle.

A request travels through clear stages, each handled by a different role:

Submit  →  Verify  →  Book engagement  →  Review decision  →  Sign-off  →  Accept / Endorse  →  Letter
(Requestor)  (Verifier)   (Requestor/Reviewer)   (Reviewer)      (Signatories)     (HOC)          (Reviewer)
How to use this guide Work top-to-bottom — the sections follow the order a request actually moves through the system. For each screen you'll find where to go, what you'll see, what to do, and what to verify. The test checklist at the end bundles it into a runnable pass.

Two channels

GCPC

Committee requests

9 matter types (RTP, PBL, JVP, ST/SP, CAA, PCCA, PP, VAP, Others). Conclude with an Endorsement letter.

GCP

Group requests

5 matter types (R-PCCA, R-PP, CI, CPR, Others). Conclude with an Acknowledgement letter.

01Roles & access

What you can see and do depends on the web role(s) assigned to your account. To test the full lifecycle you'll need accounts (or one account holding several roles) covering each stage.

RoleWhat they do in the portal
RequestorSubmits new requests. Only role that can create a request. Books engagements.
VerifierChecks a submitted request's data & documents, then advances or returns it.
ReviewerRuns the engagement review and records the decision (conclusion code).
HOCHead of Company — signs the Acceptance (GCP) or Endorsement (GCPC).
Endorser / Working GCPCGCPC-side handling and endorsement steps.
Main CommitteeRead-only view of all requests and the dashboard.
AdministratorsManage slots, engagements, signatory groups, email templates and user roles. Sees the Dashboard and Admin hub.
Tester note The portal hides links and buttons by role, but real data access is enforced by Dataverse permissions on the server. If a link is hidden, the role isn't assigned — fix it under User Role Management. After a role change the user may need to sign out and back in for it to take effect.

041 · Submit a request

Role: Requestor

The starting point of every lifecycle. A requestor picks a matter type and completes a guided multi-step form.

Choosing a request type

/submit

The page is headed "Create a Request — Select the request type to begin." with two channel tabs and a search box:

  • GCPC Requests and GCP Requests tabs switch the list of cards below.
  • Each card shows the full matter name and its short form code (e.g. PBL).
  • The search box filters cards by name or code (case-insensitive) — try typing tender or RTP.
  • Clicking a card opens the form at /submit/<channel>/<formCode> — e.g. /submit/gcpc/rtp.

The full list of types is in the Matter types reference.

Filling the multi-step form

/submit/:channel/:formCode

What you'll see

  • A step indicator / progress bar (e.g. "Step 1 of 3") with named steps such as Basic Information → Project Details → Document.
  • Back and Next buttons; Next becomes Submit Request on the last step.
  • The first step auto-fills read-only fields from your profile: Matter / title, Category (GCP/GCPC), Requestor name, Requestor email, Company.
  • Form-specific steps follow — e.g. PBL has a Bidders step where a Justification field becomes required if fewer than 3 bidders are added.
  • An Attachments uploader (files go to SharePoint as you add them) and a required acknowledgement checkbox: "I confirm the information provided is accurate."

What to do & verify

  1. Click Next with a required field empty → an inline error appears (e.g. "Field is required.") and you can't advance. Fix it and continue.
  2. Type into the form, then refresh the page within the same session → your entries are restored (draft auto-save). (Attached files are not restored — re-attach them.)
  3. Add fewer than 3 bidders on PBL → the Justification field becomes required. Add an attachment and watch it upload.
  4. Try to submit with the acknowledgement unchecked → submission is blocked ("You must acknowledge before submitting."). Tick it.
  5. Click Submit Request → button shows "Submitting…", then a success screen appears.
Success screen You should see "Request submitted" with a prominent Request ID / reference number and a View my requests button. The new request appears in the list at status New. If the notification email can't be sent you'll see a soft warning, but the request still saves.
Access check Open /submit as a non-Requestor → you're redirected to /requests. Open it signed-out → you're sent to /login.

063 · Request detail

Layout

The first action in the lifecycle. The verifier reviews the submission and chooses the next step.

What you'll see

  • Request Status dropdown — the outcome options depend on the matter type (below).
  • Comment text area — an optional note saved as the verifier comment (shown later in the detail page's General Review section).

What to do & verify

  1. Pick an outcome (e.g. Ready for Engagement), add a comment, and submit.
  2. The page polls until Dataverse confirms the new status, then returns to the detail page.
  3. Confirm the status pill changed and your comment + name now show in the General Review section.
  4. Test the branch outcomes too — FR (returned for revision) and RS (resubmission) — and confirm the list/detail reflect them.

085 · Slots (admin)

Role: Administrator   /admin/slots

Before anyone can book an engagement, an admin must publish bookable time slots. Page: "Slot Management — Create and manage meeting and engagement slots."

Tabs

Slots are split into Available and Booked tabs, with a header badge like "3 available · 1 booked".

Creating a slot

Click Create new slot to open the modal:

  • Title / type (required) — e.g. "Vendor onboarding review".
  • Start date & time (required) — must be in the future.
  • Duration (required) — chips: 15 30 45 60 min; the end time auto-calculates ("Ends at: …").
  • Reviewers (required) — a multiselect dropdown listing users who hold the Reviewer web role. Pick up to three; remove a selected reviewer with the × on its chip.

Click Create slot → "Slot created successfully and is now available for booking." New slots start as Available.

Editing

Each slot card has an Edit button. The edit modal adds a Status dropdown (Available / Booked) so you can move a slot between tabs, plus the same time/duration/reviewer fields. The Reviewers multiselect is prefilled with the slot's current reviewers (any who no longer hold the role stay selectable so you don't lose them). Save → "Slot updated successfully."

What to verify

  • A created slot appears in Available and becomes selectable in the engagement booking flow.
  • Only future-dated slots are accepted; only slots within the next ~6 months are offered for booking.
  • Each card shows date, status badge, time range + duration, title and attendee avatars.
  • The Reviewers dropdown is populated from users assigned the Reviewer web role (under User Roles). If it's empty, assign that role first; the option list stops accepting picks once three are selected.

096 · Book an engagement

Requestor / Reviewer   /requests/:id/engagement

Reached from the Book Engagement action when a request is Ready for Engagement. Schedules the meeting where the review takes place.

What you'll see

If engagements already exist, they're listed first (name + reference like R01, type badge, status badge, date/time, location, and a Cancel button on any Scheduled one). A Create Engagement button opens the booking form — unless one is already scheduled.

Booking form

  1. Engagement Name (required). The heading auto-numbers the session — e.g. "Schedule Engagement R01".
  2. Select Month — months that have available slots, each showing a count (e.g. "May 2026 (3 slots available)").
  3. Available Time Slots — a grid of slot buttons (date, duration, time range). Pick one; it highlights.
  4. Engagement Type — Virtual (default) or In-Person.
  5. Meeting Location (In-Person only) — O3CS / Hyrangea / Petunia meeting room, or Other (enter manually) which reveals a custom location field.
  6. Click Schedule Engagement.
On submit The engagement is created (Scheduled), the chosen slot flips to Booked, the request advances to R (Under Review), and you're redirected back to the detail page after a brief "Engagement scheduled" confirmation.
One-at-a-time rule A request may have only one Scheduled engagement at a time. While one is open, the create form is hidden and you see: "Engagement already scheduled. Please complete or cancel the existing scheduled engagement before creating a new one."

Cancelling

Cancel on a Scheduled engagement opens a confirm dialog. On Yes, cancel engagement the engagement becomes Cancelled and its slot is freed back to Available (if the start time is more than ~2 hours away).

107 · Engagement management (admin)

Role: Administrator   /admin/engagements

A cross-organisation view of every engagement. Page: "Engagement Management — Track and manage engagements across the group organisation."

  • Tabs group engagements by status: Scheduled (default) · Completed · Cancelled. Header badge e.g. "2 scheduled · 7 total".
  • Scheduled cards offer Manage (change type/location, or reschedule to another slot) and Cancel.
  • Cancelling frees the slot back to Available; Completed and Cancelled cards have no actions.

Engagement statuses

Scheduled → Cancelled (slot freed) or Completed (auto-set when the review is submitted — see next section).

118 · Review request

Role: Reviewer   /requests/:id/review

After the engagement, the reviewer records the formal decision. This is the core review screen, opened by the Review Request action while status is R.

Fields

Decision Code (required) — pick one radio option; the description shows beside it:

CodeLabelMeaning
Code 1ProceedProceed with acceptance of review.
Code 2ResubmitResubmit — critical information missing.
Code 3Non-compliantNo submission for review. Non-compliance.
Code 4ExemptedExempted & approved by Main GCPC (signed letter attached).
Code WWaivedWaived — signed Waiver Form from Group CEO. (ST/SP only)

Reviewer Comments (optional) — a block editor: add Text, Bulleted list or Numbered list blocks, reorder with arrows, remove with the trash icon. For PCCA / PP / R-PCCA / R-PP, an extra Info and Criteria for Review textarea appears (prefilled, editable).

What to do & verify

  1. Select a decision code and add a comment block.
  2. Click Submit Review → a confirmation modal appears. Click Confirm.
  3. The decision, comments, review date and reviewer name save; the request status & outcome update per the code (see decision codes).
  4. The linked Scheduled engagement is automatically marked Completed.
  5. The page polls for the new status, then returns to detail.

Code 1 moves the request to Draft Review; Code 2 → RS; Code 3 → NC3; Code 4 → NC4; Code W → W.

129 · Signatures (sign-off)

Roles: Signatory members

When a request reaches the Pending Review stage, a signature grid appears on the request detail page. Members of the Prepared and Confirmed signatory groups add their digital signatures.

  • Signatories are configured by an admin under Signatory Groups.
  • Each group has a minimum signature threshold (default: Prepared 1, Confirmed 2). The request can only advance to Complete Review once both minimums are met.
  • Each signatory clicks to add their signature; counts per group are checked against the thresholds.
Verify With fewer than the minimum signatures, the request cannot proceed. After enough signatures from both groups, the request can move to Complete Review and the Accept / Endorse action becomes available.

1310 · Accept (GCP) & Endorse (GCPC)

Role: HOC

The Head of Company formally accepts (GCP) or endorses (GCPC) a completed review by selecting a conclusion code and signing. Only the HOC of the requesting company may sign.

GCP channel

HOC Acceptance

/requests/:id/hoc-acceptance

Card "Review Acceptance", button Submit Acceptance. Advances to Pending Ack.

GCPC channel

Endorsement

/requests/:id/endorse

Card "Review Endorsement", button Submit Endorsement. Advances to Pending Endorse.

Conclusion codes

CodeMeaning
Code 1 (a)Agree to incorporate all comments in submission / future action.
Code 1 (b)Agree to incorporate all comments except listed exceptions (3 exception/mitigation fields appear).
Code 2Acknowledge the need to resubmit and incorporate comments.
Code 3Acknowledged — admit the non-compliance.

What to do & verify

  1. Select a conclusion code (if 1(b), fill the exception fields).
  2. Click Add Signature and provide the digital signature (the box shows "Awaiting signature" until then).
  3. Click Submit Acceptance / Endorsement → confirm in the modal.
  4. Verify: once signed, the whole form locks (read-only) and shows "Signed: dd/mm/yyyy"; the request status advances.
  5. Sign in as an HOC from a different company → you should be blocked with a company-mismatch warning.

A Print button is available in the header.

1411 · Letters

After acceptance/endorsement, the reviewer generates the closing letter from a matter-specific template with auto-filled and manual fields.

LetterChannelPathResult status
AcknowledgementGCP/requests/:id/ack-letterACK
EndorsementGCPC/requests/:id/endorsement-letterE

How it works

  • The template is chosen automatically from the matter type. It includes a header (with an auto-generated document number), an info table, body paragraphs, sign-off, attachments list and a CC list.
  • Auto-filled fields (company, project, dates, matter label) are highlighted and read-only.
  • Manual fields (e.g. Review No., contract values, package counts) are editable inputs — left blank, they appear as «placeholders».
  • Toggle Edit ⇄ Preview; Print opens the browser print dialog.

What to do & verify

  1. Fill the manual fields, switch to Preview, then click Submit Acknowledgement / Endorsement.
  2. If any manual field is empty, a soft warning lists them — you can go back or submit anyway.
  3. Confirm → the letter is saved, the request advances to ACK / E, and you're returned to the detail page.
  4. Re-open the letter → it loads read-only with your values restored, and an Edit button lets you reopen it.

15Signatory groups (admin)

Role: Administrator   /admin/signatories

Defines who signs requests and the minimum signatures required. Members are split into Prepared and Confirmed groups.

Adding a member

  1. Pick a Group (Prepared / Confirmed).
  2. Start typing a Name — matching contacts appear; pick one to auto-fill the email (or type it).
  3. Click Add member — they appear in that group's table.

Remove a member with the per-row Remove button (confirm dialog). To change details, remove and re-add.

Thresholds

The Minimum signature thresholds card sets Prepared (minimum) and Confirmed (minimum); Save thresholds → "Thresholds saved." These gate the sign-off step.

16Email templates (admin)

Role: Administrator   /admin/email-templates

Controls the notification emails sent at each lifecycle milestone. One template per event × recipient role, each independently toggled on/off.

Events & recipients

Events: Request Submitted, Request Verified, Request Reviewed. Recipients: Requester, Verifier, Reviewer, Working GCPC, Endorser, HOC.

Editing a template

  1. Pick an event + role from the Template dropdown. A badge shows Not set / On / Off.
  2. Edit the Subject and build the body from blocks: Heading, Paragraph, Info Rows, Button, Divider.
  3. Insert placeholder tokens (e.g. {{requestName}}, {{requestNumber}}, {{viewLink}}) by clicking a chip — it drops into the focused field.
  4. Tick Active (send this email) to enable sending.
  5. Watch the Live Preview (right side) render with sample data, then click Save template → "Template saved."
Verify The footer toggles between "Unsaved changes" and "All changes saved"; Discard reverts to the last save. With a template Active, performing the matching action (submit / verify / review) should trigger the email. A Help button explains tokens and blocks in detail.

17User role management (admin)

Role: Administrator   /admin/user-roles

Assign or remove portal web roles per contact. Title: "User Role Management — Select a user, set their portal web roles, then save your changes."

  • Non-admins see "Administrators only. You need the Administrators web role to manage user roles." — and the server re-checks admin rights regardless of the UI.
  • Left: search users by name/email; each row shows avatar, name, email; a counter shows "X of Y shown".
  • Right: tick/untick role checkboxes (Administrators, Requestor, Reviewer, Verifier, HOC, Working GCPC, Endorser, Main Committee, …). Changed roles are highlighted.
  • Footer reads "Unsaved changes" / "All changes saved"; Save changes applies the diff → "Roles saved."; Discard reverts.
Verify Switching to another user with unsaved edits warns you first. After saving, the target user may need to sign out / in for new permissions to apply. Assign a Requestor role to a test account and confirm the Submit flow then opens for them.

R5End-to-end test checklist

A happy-path pass that exercises the whole portal. Use accounts covering Requestor, Verifier, Reviewer, HOC and Administrator.

Setup (Administrator)

  • Sign in; confirm Dashboard and Admin links appear.
  • Under User Roles, assign Requestor / Verifier / Reviewer / HOC to your test accounts.
  • Under Slots, create at least one future Available slot.
  • Under Signatory Groups, add Prepared & Confirmed members and set thresholds.
  • Under Email Templates, set a Submitted/Verified/Reviewed template Active and Save.

Submit (Requestor)

  • Open /submit; pick a GCPC matter (e.g. PBL). Confirm read-only profile fields.
  • Trigger a validation error, then a draft-restore on refresh.
  • Attach a file, tick acknowledgement, submit → capture the Request ID.
  • Confirm the request shows as New in the list.

Verify → Engage → Review

  • (Verifier) Open the request → Verify Data → set Ready for Engagement + comment.
  • (Requestor/Reviewer) Book Engagement → pick the slot; confirm slot becomes Booked & status → R.
  • (Reviewer) Review Request → Code 1 + comments → Confirm; engagement auto-Completed, status → Draft Review.

Sign-off → Accept/Endorse → Letter

  • (Signatories) At Pending Review, add signatures until thresholds met → status → Complete Review.
  • (HOC) Endorse (GCPC) / Accept (GCP): choose a conclusion code, sign, submit. Confirm company-mismatch is blocked.
  • Generate the letter: fill manual fields, preview, submit → status → E / ACK.
  • (Admin) Confirm the Dashboard counts updated, and a notification email fired for active templates.

Branch paths to spot-check

  • Verify with FR and RS outcomes and confirm the list/detail reflect them.
  • Review with Code 2 / 3 / 4 and confirm the resulting RS / NC3 / NC4 statuses.
  • Try to book a second engagement while one is Scheduled → blocked. Cancel one → slot freed.
Logging issues For each failure capture: the page URL, the request ID, your role(s), what you expected vs. what happened, and a screenshot. That makes every bug reproducible.
